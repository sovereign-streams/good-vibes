Stream Exchange Protocol (SEP)

Version 0.1.0 — An open protocol for exchanging metadata-driven content between sovereign providers and consumers.

1 Overview

The Stream Exchange Protocol (SEP) defines how Stream Providers and Stream Consumers exchange metadata-driven content through structured negotiation. SEP is content-agnostic — it defines the exchange format, not the content type. A stream could be video metadata, podcast metadata, article summaries, music tracks, dashboard data, or any structured content.

The protocol doesn't care what flows through it — only how it's described and negotiated.

Protocol version: 0.1.0 — This is a draft specification. Breaking changes are expected before 1.0.0.

SEP defines five message types:

  1. Enrichment Envelope — Semantically tagged content metadata
  2. Consumer Intent — What the consumer is looking for
  3. Provider Response — Curated payload of enriched items
  4. Provider Manifest — Provider capabilities and handshake
  5. Telemetry Exchange — Optional engagement metrics

2 Design Philosophy

SEP is built on several foundational principles:

  • Consumer sovereignty — The consumer always controls what it shares and how content is filtered
  • Provider transparency — Providers publish their curation standards openly
  • Metadata exchange — Streams exchange metadata, not full media. Full media is lazy-loaded by the consumer
  • Content agnosticism — The protocol has no opinion about content. Only format.
  • Privacy by default — State lives on the consumer side. Providers see only what is explicitly shared.
  • Lightweight and self-hostable — No dependency on any platform's infrastructure

3 Meta-First Principle

SEP payloads contain metadata, not full media. A SEP payload contains:

  • Semantic tags (categories, emotional tone, scores)
  • Source pointers (URLs to origin content)
  • Enrichment scores (energy, cognitive load, motivation, etc.)
  • Provider confidence ratings

Full media is lazy-loaded by the consumer only when needed. This keeps the protocol lightweight and cost-effective. You're indexing meaning, not storing video blobs.

5 Consumer Intent Schema

When a consumer queries a provider, it sends an intent payload describing what it's looking for. The consumer controls how much information it reveals to the provider.

Consumer Intent 
{
  "sep_version": "0.1.0",
  "consumer_id": "anonymous-or-identified",
  "intent": {
    "session_type": "composed",
    "target_duration_minutes": 15,
    "weights": {
      "fitness": 0.25,
      "humor": 0.20,
      "skill_building": 0.20,
      "motivation": 0.15,
      "craft": 0.10,
      "music": 0.10
    },
    "filters": {
      "exclude_rage_bait": true,
      "exclude_humiliation": true,
      "exclude_shock_content": true,
      "min_energy_level": 0.2,
      "max_cognitive_load": 0.7,
      "language": ["en"]
    },
    "context": {
      "time_of_day": "morning",
      "session_number_today": 1,
      "state_token": null
    }
  },
  "disclosure_level": "minimal",
  "telemetry_opt_in": false
}

Disclosure Levels

The consumer always chooses its disclosure level. The provider works with what it gets.

Level What the Provider Sees Use Case
minimal Only weights and filters. No behavioral history. Maximum privacy. Good for first-time queries.
standard Weights, filters, plus recent session summaries (anonymized). Better personalization with moderate privacy.
full Weights, filters, full behavioral profile. Maximum personalization. Provider can hyper-personalize.

The provider responds with a curated payload of enriched items, along with metadata about the response quality and provider capabilities.

        "meta": { "title": "Why Stoics Wake Up at 5 AM" },

    "confidence": 0.78,
    "suggested_arc": [

7 Provider Manifest Schema

The provider manifest is the handshake document. It declares the provider's identity, capabilities, supported exchange models, guardrails, and rate limits. Both sides use this to negotiate the exchange.

Provider Manifest 
{
  "sep_version": "0.1.0",
  "provider_id": "good-vibes-main",
  "provider_name": "Good Vibes",
  "description": "Cognitive environment designer for positive content consumption",
  "endpoint": "https://api.goodvibes.app/sep/v1",
  "supported_models": ["negotiated", "curated_payload"],
  "supported_content_types": ["video", "podcast", "music", "article"],
  "guardrails": {
    "published": true,
    "url": "https://goodvibes.app/guardrails",
    "version": "0.1.0"
  },
  "enrichment_schema_version": "0.1.0",
  "max_payload_size": 100,
  "rate_limit": {
    "requests_per_minute": 30,
    "daily_cap": 1000
  }
}

8 Telemetry Exchange Schema

If the consumer opts in, it can share engagement metrics back to the provider. This creates a value exchange: the provider gets signal quality data, the consumer gets better future payloads. Telemetry is never required.

Telemetry Exchange 
{
  "sep_version": "0.1.0",
  "telemetry": {
    "session_id": "uuid",
    "items": [
      {
        "item_id": "uuid",
        "viewed": true,
        "view_duration_seconds": 45,
        "completed": false,
        "liked": false,
        "skipped_at_seconds": 45,
        "rewatched": false,
        "paused": true,
        "paused_at_seconds": 22
      }
    ],
    "session_completed": true,
    "session_satisfaction": 0.8
  }
}

Privacy note: Telemetry is aggregated before sending (if opted in). No behavioral fingerprinting from telemetry data. The consumer controls the granularity.

9 Exchange Models

The protocol supports multiple exchange patterns. Both sides declare what they support during the initial manifest handshake.

Model A: Provider-Curated Payload

Consumer sends intent. Provider filters heavily and returns a curated set. This is the simplest model. The provider has more influence over results.

Model B: Full Meta-Index Browse

Provider exposes its entire metadata index. Consumer queries directly against it, like a search engine. This offers the most consumer sovereignty but is the most expensive for the provider. The provider can observe query patterns.

Model C: Negotiated Exchange (Recommended Default)

Consumer sends intent with chosen disclosure level. Provider returns a ranked payload with confidence scores and alternative clusters. Consumer refines locally using its PAE. Multiple round-trips are expected. State tokens enable continuity without the provider storing user profiles.

Recommended: Model C (Negotiated Exchange) is the recommended default. It balances consumer sovereignty with provider efficiency and supports progressive refinement through state tokens.

10 Privacy Architecture

Core rule: State lives primarily on the consumer side.

The provider sees only what the consumer chooses to share. The consumer's PAE does the heavy personalization work locally. The provider's job is to return a good-enough payload — not to build a user profile.

Privacy-Preserving Measures

  • Disclosure levels control what the provider sees
  • State tokens are opaque — the provider stores session continuity, not behavioral profiles
  • Telemetry is aggregated before sending (if opted in)
  • No behavioral fingerprinting from intent vectors — consumers can rotate or generalize their weight vectors
  • The consumer can query multiple providers with different disclosure levels

11 Versioning Strategy

SEP uses dual versioning:

Semantic Versioning (Protocol)

The protocol version follows semver: 0.1.0, 0.2.0, 1.0.0. Breaking changes increment the major version. New features increment the minor version. Bug fixes increment the patch version.

Date-Based Snapshots (Schema)

Schema releases use date-based snapshots: spec/2026-02-28/. This allows for reproducible schema definitions pinned to a specific release date.

Enrichment Schema Versioning

Enrichment schemas are versioned independently from the protocol. A consumer can request items enriched at a minimum schema version. Old enrichments remain valid at their declared version.

  • Every enriched record carries enrichment.schema_version
  • New queries can specify min_schema_version
  • Re-enrichment is done on-access (lazy) or via batch script
  • Breaking changes = major, new dimensions = minor, fixes = patch
